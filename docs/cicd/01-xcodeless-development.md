# 01 — Xcode-less development: CI is the compiler

This repo is developed in remote Claude Code sessions on Linux. There is no
Xcode, no simulator, no `swift build` that can compile an iOS app. That
constraint drives every infrastructure decision below — and it turns out to
produce a setup that is *also* better for teams that do have Macs: the
project is plain text, the build is reproducible, and "works on my machine"
cannot happen because there is no machine.

## The core loop

1. Edit Swift sources and `project.yml` as plain text.
2. Push to a branch matching the CI trigger (`main` or `claude/**` here).
3. `ios-build.yml` generates the Xcode project, builds, and runs the full
   test suite against a simulator.
4. Read the log (via `xcbeautify --renderer github-actions`, errors surface
   as inline annotations), fix, push again.

That's the whole development experience. A green check means "compiles and
all tests pass"; there is no other definition of working.

## XcodeGen: the project is plain text

The `.xcodeproj` is **generated and gitignored**. The single source of
truth is [`ios/project.yml`](../../ios/project.yml), an
[XcodeGen](https://github.com/yonaskolb/XcodeGen) spec declaring targets,
SPM packages, build settings, and the test scheme.

Why this matters:

- **Editable without Xcode.** Adding a source file, an SPM dependency, or a
  build setting is a text edit, reviewable in a PR diff. No
  `project.pbxproj` merge conflicts, ever.
- **Sources are globbed.** `sources: [Sources]` means new `.swift` files
  are picked up automatically — creating a file *is* adding it to the
  target.
- **CI and local Xcode agree.** Anyone with a Mac runs
  `brew install xcodegen && cd ios && xcodegen generate && open *.xcodeproj`
  and gets exactly the project CI builds.

The corresponding `.gitignore` entry:

```gitignore
## Generated by XcodeGen — regenerate with `xcodegen generate` in ios/
ios/*.xcodeproj
```

## Pin the runner, float the point release

Because CI is the only compiler, an unexplained toolchain change is
indistinguishable from a broken commit. So:

- **The runner image is pinned** (`macos-26`, never `macos-latest`). A
  floating alias flipping to a new image swaps the Xcode/SDK/simulator set
  under you with no repo change to explain the breakage — and with no local
  Xcode, no way to diagnose it interactively.
- **Within the image, select the newest matching Xcode major** and print
  it, so minor image updates don't require a repo change:

  ```bash
  XCODE_APP="$(ls -d /Applications/Xcode_26*.app | sort -V | tail -1)"
  sudo xcode-select -s "$XCODE_APP/Contents/Developer"
  xcodebuild -version
  ```

- **Runner bumps are deliberate, isolated commits** made when GitHub's
  deprecation warnings appear — never mixed with feature work, so a
  toolchain-caused failure bisects to exactly one commit.

## Design consequences for the app code

Working CI-only shapes what "testable" means:

- **Logic that matters lives in plain Swift** (here: `ios/Sources/Core/`),
  unit-tested with no UI. State machines use an architecture with a
  headless test harness (TCA `TestStore` here) so behavior is exhaustively
  testable without driving a simulator UI.
- **Network flows are tested against an in-process localhost mock server**
  (FlyingFox serving scripted SSE), so integration tests run inside the
  normal `xcodebuild test` job. This needs `NSAllowsLocalNetworking` in
  Info.plist (see [gotchas](06-gotchas.md)).
- **Anything only verifiable by a human on a device** (gestures, rendering)
  ships via the distribution workflows ([doc 04](04-firebase-distribution.md))
  and gets checked on a real phone.

## What a new repo needs to adopt this

- `ios/project.yml` (or wherever your project root is) as the XcodeGen
  spec; `.xcodeproj` gitignored. If the repo has an existing checked-in
  `.xcodeproj`, converting is optional — the CI workflow works either way,
  you just skip the `xcodegen generate` step — but you lose the plain-text
  benefits.
- The build & test workflow from [doc 02](02-build-and-test.md).
- A README note telling humans how to get a local Xcode project
  (`brew install xcodegen; xcodegen generate`).

The [`ios-ci-setup`](../../.claude/skills/ios-ci-setup/SKILL.md) skill
automates this.
